The self-auditing loop — spelled out

the agents fact-checked the build and caught real errors

Here is the loop, in plain terms — this is the part that is genuinely different from “a developer using AI autocomplete”:

  1. The build started from a written baseline — eligibility rules and platform behavior drafted from domain knowledge (the equivalent of your BA's first-pass requirements doc).
  2. Seven research agents were sent to verify it independently against primary sources: CDSS ACINs, the CalSAWS Configuration Guide, county handbooks, the live public portal.
  3. Where a source contradicted the baseline, the source won. Each discrepancy came back with a citation, the engine parameters were corrected, and the automated tests were re-pinned to the corrected math — same day.
  4. The QA harness then re-proved everything — 33 checks, including the exact dollar outcomes.

One example, walked end to end

Maria's CalFresh allotment (family of 3 · $1,600/mo earnings · $1,400 rent).
① Baseline computed with FFY2025 values (max allotment $768, standard deduction $204, shelter cap $712) → benefit $658/mo.
② The Policy I agent pulled CDSS ACIN I-46-25 (effective 10/2025) and found the FFY2026 values: max allotment $785, standard deduction $209, shelter cap $744, SUA $663.
③ The engine's parameters were corrected and the unit test was re-pinned: Maria's benefit is $686/mo.
④ Why you should care: shipped with the stale numbers, every CalFresh determination in the system would have been wrong — for this family, by $28/month. The error was caught by an agent, with a citation, in the same working session. No release train, no regression sprint.

What this means for a real program

  • Annual COLA updates become a config change + an automated regression run — hours, not a release cycle. (Watch it live in the demo: Admin flips a rule, the next determination uses it.)
  • Policy drift gets caught by agents with citations — the RCA change is exactly the kind of mid-year federal change that silently breaks eligibility systems.
  • Every determination carries its own audit trail — the trace stored with each EDBC result shows the tests, deductions, and dollar math a hearing officer would ask for.

System topology

who talks to what, with which key
auth flow (anon key) RLS-scoped reads (user JWT) engine writes + atomic RPCs (server-only)

Two keys, two trust levels: the browser only ever holds the anon key, so every direct query is row-filtered by RLS. The service-role key exists only inside server actions. The atomic workflow RPCs are called with the user's JWT, so auth.uid() inside the database identifies the true actor — authorization is enforced in Postgres even if the app layer were bypassed.

The case lifecycle

state machine + what each transition emits
The atomic step. _finalize_run() is one SQL transaction that writes the NOAs (per program, plain-language reasons), the EBT issuances (eligible + amount > 0), flips the case status, closes open tasks, and appends the journal entry. A half-issued benefit is impossible — either everything commits or nothing does.

Data model

15 tables · cases as the hub

Left side is the input snapshot the engine consumes; right side is everything a determination emits. Traces are stored as jsonb on edbc_results, so any past determination can be re-explained, audited, or fed to the Copilot verbatim.

Role escalation is impossible from the client: roles live in JWT app_metadata, writable only with the service key. Self-signup always lands as applicant. e2e scenario 4 proves the crit cells: an applicant session selecting another applicant's cases gets zero rows, not an error.

The EDBC engine

six programs · FFY2026-verified · fully traced
